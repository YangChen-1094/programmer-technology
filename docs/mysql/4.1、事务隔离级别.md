### **1、mysql的四种隔离级别**

<span style="color:red">读未提交</span>（read uncommitted）：指一个事务还没提交时，它做的变更就能被其他事务看到；

<span style="color:red">读提交</span>（read committed）：指一个事务提交之后，它做的变更才能被其他事务看到；

<span style="color:red">可重复读</span>（repeatable read）：指一个事务执行过程中看到的数据，一直跟这个事务启动时看到的数据是一致的，MySQL InnoDB 引擎的默认隔离级别；

<span style="color:red">串行化</span>（serializable ）：会对记录加上读写锁，在多个事务对这条记录进行读写操作时，如果发生了读写冲突的时候，后访问的事务必须等前一个事务执行完成，才能继续执行；

![图片](images/4.1、事务隔离级别/img1.jpg)

当多个事务并发执行时可能会遇到「<span style="color:red">脏读</span>、<span style="color:red">不<span style="color:red">可重复读</span></span>、<span style="color:red">幻读</span>」的现象，这些现象会对事务的一致性产生不同程序的影响。

<span style="color:red">脏读</span>：读到其他事务未提交的数据；

<span style="color:red">不<span style="color:red">可重复读</span></span>：前后读取的数据不一致；

<span style="color:red">幻读</span>：前后读取的记录数量不一致。

这三个现象的严重性排序如下：<span style="color:red">脏读</span>  > <span style="color:red">不<span style="color:red">可重复读</span></span> > <span style="color:red">幻读</span>


### **2、不同隔离级别的实现**

#### <span style="color:red">2.1、可<span style="color:red">串行化</span>（Serializable）</span>

在可<span style="color:red">串行化</span>级别上，MySQL执行S2PL并发控制协议, 一阶段申请，一阶段释放。读写都要加锁。使用到的锁有：

共享锁（S Lock）：用于读取数据。

排他锁（X Lock）：用于更新数据。

间隙锁（Gap Locks）：用于防止其他事务在当前事务读取的范围内插入新数据。

记录锁（Record Locks）：锁定特定的行。

临键锁（Next-Key Locks）：锁定记录及其前面的间隙。

意向锁（Intention Locks）：用于指示事务的锁定意图。

#### <span style="color:red">2.2、<span style="color:red">可重复读</span>（Repeatable Read）</span>

<span style="color:red">可重复读</span>是MySQL默认的隔离级别，理论上说应该称作快照（Snapshot）隔离级别。**<span style="color:red">读不加锁，只有写才加锁，读写互不阻塞</span>**，并发度相对于可<span style="color:red">串行化</span>级别要高，但会有Write Skew异常。

事务在开始时创建一个ReadView，当读一条记录时，会遍历版本链表，通过当前事务的ReadView判断可见性，找到第一个对当前事务可见的版本，读这个版本。对于写操作，包括Locking Read(SELECT ... FOR UPDATE), UPDATE, DELETE，需要加写锁。根据谓词条件上索引使用情形，锁定有不同的方式：
* 有索引：对于索引上有唯一约束且为等值条件的情形，不用GAP LOCK，只锁定索引记录。对于其它情形，使用GAP LOCK，相当于谓词锁。
* 没有索引：由于MySQL没有实现通用的谓词锁，这时就相当于锁全表。

使用到的锁有：
* 共享锁（S Lock）：用于读取数据。
* 排他锁（X Lock）：用于更新数据。
* 间隙锁（Gap Locks）：用于防止其他事务在当前事务读取的范围内插入新数据。
* 记录锁（Record Locks）：锁定特定的行。
* 临键锁（Next-Key Locks）：锁定记录及其前面的间隙。
* 意向锁（Intention Locks）：用于指示事务的锁定意图。

#### <span style="color:red">2.3、读已提交（Read Committed）</span>

MySQL的读已提交实际是语句级别快照。与<span style="color:red">可重复读</span>级别主要有两点不同：

1）获得ReadView的时机。每个语句开始执行时，获得最新的ReadView。

2）写锁的使用方式。这里不需要GAP LOCK，只使用记录锁。并且事务只持有被UPDATE/DELETE记录的写锁（<span style="color:red">可重复读</span>需要保留全部写锁直到事务结束，而读已提交只保留真正更改的）。

使用到的锁有：
* 共享锁（S Lock）：用于读取数据。
* 排他锁（X Lock）：用于更新数据。

<span style="color:red">2.4、<span style="color:red">读未提交</span>（Read Uncommitted）</span>
读最新的数据，不管这条记录是不是已提交。不会遍历版本链，少了查找可见的版本的步骤。这样可能会导致<span style="color:red">脏读</span>。
* 排他锁（X Lock）：当一个事务进行写操作（如 INSERT、UPDATE 或 DELETE）时，MySQL 会施加排他锁。这个锁会阻止其他事务对同一行进行写入和读取，直到当前事务提交或回滚。
