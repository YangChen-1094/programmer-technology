### 1、 **count(1)**、**count(*)**、count(主键)，count(字段)的性能排序

**count(1)** = **count(*)**  > count(主键) > count(字段)

### 2、count(主键字段) 执行过程是怎样的？

2.1、如果表里只有主键索引，没有**二级索引**时。

**InnoDB** 循环遍历**聚簇索引**，将读取到的记录返回给 server 层，然后读取记录中的 id 值，就会 id 值判断是否为 NULL，如果不为 NULL，就将 count 变量加 1。

2.2、如果表里有**二级索引**时。

**InnoDB** 循环遍历的对象就不是**聚簇索引**，而是**二级索引**。读取**二级索引**中的值，就会根据值判断是否为 NULL，如果不为 NULL，就将 count 变量加 1。（因为相同数量的**二级索引**记录可以比**聚簇索引**记录占用更少的存储空间，所以**二级索引**树比**聚簇索引**树小，这样遍历**二级索引**的 I/O 成本比遍历**聚簇索引**的 I/O 成本小，因此「优化器」优先选择的是**二级索引**。）

### 3、**count(1)** 执行过程是怎样的？

3.1、如果表里只有主键索引，没有**二级索引**时。

**InnoDB** 循环遍历**聚簇索引**（主键索引），将读取到的记录返回给 server 层，但是不会读取记录中的任何字段的值，因为 count 函数的参数是 1，<span style="color:red">不是字段，所以不需要读取记录中的字段值</span>。参数 1 很明显并不是 NULL，因此 server 层每从 **InnoDB** 读取到一条记录，就将 count 变量加 1。（**count(1)** 相比 count(主键字段) 少一个步骤，就是不需要读取记录中的字段值，所以通常会说 **count(1)** 执行效率会比 count(主键字段) 高一点）。

3.2、如果表里有**二级索引**时, **InnoDB** 循环遍历的对象就**二级索引**了。

### 4、**count(*)** 执行过程是怎样的？

对于 selete * 这条语句来说是这个意思，但是在 **count(*)** 中并不是这个意思。count(\*) 其实等于 count(0)，**count(1)**，count(2)，也就是说，当你使用 **count(*)** 时，MySQL 会将 * 参数转化为参数 0 来处理。

**count(*)** 执行过程跟 **count(1)** 执行过程基本一样的，性能没有什么差异。（在 MySQL 5.7 的官方手册中有这么一句话：

**InnoDB** handles SELECT COUNT(\*) and SELECT COUNT(1) operations in the same way. There is no performance difference.

翻译：**InnoDB**以相同的方式处理SELECT COUNT（\*）和SELECT COUNT（1）操作，没有性能差异。）

而且 MySQL 会对 **count(*)** 和 **count(1)** 有个优化，如果有多个**二级索引**的时候，优化器会使用key_len 最小的**二级索引**进行扫描。只有当没有**二级索引**的时候，才会采用主键索引来进行统计。

### 5、count(字段) 执行过程是怎样的？

```sql
// name不是索引，普通字段select count(name) from t_order;
```

对于这个查询来说，会采用全表扫描的方式来计数，所以它的执行效率是比较差的。
